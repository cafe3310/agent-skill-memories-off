# 审计与日志定义: Git Audit Specification

本文档定义了 `memories-off` 中利用 Git 进行变更审计的规范。

---

## 1. 核心机制 (Core Mechanism)

- **版本控制**: 知识库根目录必须初始化为 Git 仓库。
- **强制提交**: 所有的 `Mutation` 操作（增删改）完成后，必须通过脚本或命令执行 `git commit`。
- **强制理由**: 每一个提交必须包含 `--reason` 参数，解释此次变更的动机。

## 2. 提交信息规范 (Commit Message Standard)

为了确保审计信息的完整性，系统及 Agent 应遵循以下提交信息格式：

`<subcommand> [Target Info]: <Reason>`

### 字段说明：
- **subcommand**: 执行的 `memocli` 子命令（如 `append-update`, `rename-entity`, `doctor`）。
- **Target Info**: 可选。受影响的实体名称或范围（如 `EntityA -> EntityB`）。
- **Reason**: 核心。由用户或 Agent 提供的执行理由。

### 示例：
- `append-update 人物-cafe3310: 更新了近期参与的项目列表`
- `create-entity 宠物-咪咪: 新增猫咪基础档案`
- `rename-entity 物品-渴望猫粮 -> 物品-Orijen: 统一使用英文原名`
- `doctor [rule-normalize-headers]: 批量纠正全库标题层级`

## 3. 自动审计逻辑

- **自动提交**: 绝大多数 `memocli` 修改指令都内置了自动 `git add .` 和 `git commit` 逻辑。
- **元数据同步**: 在修改实体内容时，系统会自动在 Frontmatter 中更新 `date modified`，并视情况将 `reason` 存入元数据（如 `rename-entity` 操作）。

## 4. 审计查询

- Agent 可通过 `run_shell_command` 执行 `git log -n 5` 来了解最近的变更历史，辅助决策。
