# 文档结构定义: Entity Structure

本文档定义了 `memories-off` 中实体文件的标准 Markdown 结构，旨在确保数据的机器可读性和人类可维护性。

---

## 1. 结构概览 (Structure Overview)

每个实体文件由两个核心部分组成：**Front Matter (元数据区)** 和 **Body (正文内容区)**。

```markdown
---
entity type: [Type]
date created: YYYY-MM-DD HH:MM:SS
date modified: YYYY-MM-DD HH:MM:SS
aliases: [Alias 1], [Alias 2]
relation as [predicate]: [Target Entity]
---

# [Entity Name] (H1)

[Content Block]

## [Section Title] (H2)

[Content Block]
```

## 2. Front Matter (元数据区)

### 2.1 规范要求
- **格式**: 必须位于文件最顶部，由一对 `---` 分隔。
- **内容**: 仅限一级键值对，不支持嵌套对象。
- **行布局**: 每个条目必须独立占据一行，格式为 `Key: Value`。

### 2.2 核心元数据 (Core Metadata)
- **`entity type`**: 指定实体的分类（如 `人物`, `概念`, `项目`）。
- **`date created`**: 记录实体的最初创建时间。
- **`date modified`**: 记录实体的最后修改时间（格式：`YYYY-MM-DD HH:MM:SS`）。
- **`aliases`**: 实体的替代名称（别名），以逗号分隔。
- **`relation as [predicate]`**: 显式定义与另一个实体的语义关系。

## 3. Body (正文内容区)

### 3.1 标题层级规范 (Heading Hierarchy)
为了确保图谱结构的扁平化与可解析性，实体正文必须遵循严格的标题契约：
- **H1 (`# `)**: **仅限首行使用**，代表实体名称。
- **H2 (`## `)**: 代表正文中的**业务章节**。所有后续添加或更新的内容块，其内部标题均应标准化为 H2。
- **禁止层级**: 禁止使用 H3 (`### `) 及更深层级的标题。系统在写入或通过 `doctor` 治理时会自动将其拍平。

### 3.2 内容组织
- **无序列表**: 建议使用 `-` 组织零散信息。
- **WikiLinks**: 使用 `[[实体名]]` 或 `[[标准名|显示名]]` 建立隐式链接。

## 4. 解析与编辑协议 (Lifecycle)

- **标准化定位**: 系统通过 `MetadataParser` 识别元数据，通过正则表达式定位章节。
- **非破坏更新**: 优先使用 `memocli append-update` 以追加 `UPDATE_BLOCK` 的形式进行修改，由系统自动维护时间戳与标题合规性。
- **健康自愈**: 周期性使用 `memocli doctor --rule-normalize-headers` 确保全库标题层级的一致性。
