# 操作定义: Management Tools

本文档定义了用于管理知识实体、类型及跨库级任务的工具规格。

---
## 1. 实体生命周期管理 (Entity Management)

### 1.1 `create-entity`
- **职能**: 创建全新的实体文件。支持一并注入初始正文内容、别名和关系。
- **[▶️执行方式]**:
  - 命令行模式: `memocli create-entity --name <name> --type <type> --content <text> --reason <reason>`
  - STDIN 模式 (推荐): 
    ```bash
    echo "## 人物志
    - 姓名: [[cafe3310]]
    - 特点: 喜欢折腾 Agent 和猫。" | memocli create-entity --name "cafe3310" --type "人物" --content-stdin --reason "新建档案"
    ```
- **执行逻辑**:
  - 1. 自动执行名称标准化。
  - 2. 注入 Frontmatter 并自动生成 `date created`。
  - 3. 内容将自动被包裹在 `UPDATE_BLOCK` 中并执行 H2 标题拍平。
  - 4. `--content` 与 `--content-stdin` 互斥。

### 1.2 `rename-entity`
- **职能**: 重命名实体文件，并自动同步全库范围内的 WikiLinks 引用及元数据关系。
- **[▶️执行方式]**: `memocli rename-entity --old <old> --new <new> --reason <reason>` **(已内置 commit)**

### 1.3 `merge-entities`
- **职能**: 将多个源实体内容合并到目标实体，并同步重定向全库引用，最后删除源文件。
- **[▶️执行方式]**: `memocli merge-entities --sources <s1,s2> --target <target> --reason <reason>` **(已内置 commit)**

### 1.4 `delete-entity` (删除)
- **职能**: 移除实体文件。由于知识库受 Git 保护，直接物理删除即可。
- **[▶️执行方式]**: `rm entities/<name>.md`，随后执行 `memocli commit --reason "删除实体: <reason>"`。

## 2. 类型管理 (Type Management)

### 2.1 `rename-type`
- **职能**: 全库范围内重命名特定的实体类型并同步 `meta.md`。
- **[▶️执行方式]**: `memocli rename-type --old <old> --new <new> --reason <reason>` **(已内置 commit)**
- **特性**: 脚本会扫描 `meta.md` 并列出除标准定义外的所有旧类型匹配行，提醒 Agent 进行手动同步。

### 2.2 `merge-types`
- **职能**: 将一个或多个源类型合并到目标类型中。
- **[▶️执行方式]**: `memocli merge-types --sources <src1,src2> --target <target> --reason <reason>` **(已内置 commit)**

## 3. 系统治理 (System Governance)

### 3.1 `doctor`
- **职能**: 知识库的一站式健康自愈工具。
- **[▶️执行方式]**: `memocli doctor --rule-<name> [--fix] --reason <reason>`
- **核心场景**: 批量标准化命名、拍平标题层级、清理死链接、修复陈旧更新块格式。

### 3.2 `init`
- **职能**: 在指定目录下初始化一个新的标准知识库。
- **[▶️执行方式]**: `memocli init --path <path> --name <kb_name>`

---

## 4. 操作准则 (Guidelines)

- **别名互补**: 在重命名实体后，`doctor` 会尝试将原名存入 `aliases` 字段，以保持图谱检索的连续性。
- **类型严谨**: 修改类型是全局性变动，在执行前应先通过 `search --type` 评估影响规模。
- **提交理由**: 所有管理工具都必须提供 `--reason`，这直接决定了 Git 历史的可读性。
