# 核心概念与数据模型 (Core Concepts & Data Model)

本文档面向 Agent，旨在简明扼要地解释 `memories-off` 知识库中的核心概念、数据结构以及你在调用 `memocli` 工具时所处理的“东西”究竟代表什么。

---

## 1. 知识库架构 (Library Architecture)

当你使用 `memocli` 操作知识库时，你面对的是一个由 Markdown 文件组成的物理文件夹：

- **`meta.md` (手册)**: 知识库的大脑。它定义了当前知识库的全局规则、允许使用的实体类型（Type）以及预设的关系谓词（Predicate）。`memocli explore` 会为你读取它的内容。
- **`entities/` (实体目录)**: 存放所有具体知识的容器。每一个 `.md` 文件就是一个**实体 (Entity)**。
- **`.git/` (审计追踪)**: 知识库受 Git 保护。你的每一次实质性修改（无论是追加内容还是重命名），底层都会自动或要求你执行一次 `commit`，并强制你附带 `--reason` 参数以记录操作动机。

## 2. 实体与元数据 (Entity & Metadata)

实体是知识的最小独立单元。每个实体文件分为两部分：顶部的 **YAML Frontmatter (元数据)** 和下方的 **Body (正文)**。

### 2.1 元数据 (Frontmatter)
调用诸如 `create-entity`, `search-entities` 等命令时，你会高频操作以下元数据字段：
- **`entity type`**: 实体的分类（如 `人物`, `项目`）。必须是单值。
- **`aliases`**: 实体的别名列表（逗号分隔）。这非常重要，`search-entities` 会利用别名进行智能回退查找。
- **`date created` / `date modified`**: 系统自动维护的时间戳。

### 2.2 正文结构约束 (H1 & H2 Rules)
为了确保你能通过 `memocli` 准确地追加和读取内容，正文被施加了严格的**层级约束**：
- **H1 (`# `)**: **仅限文件首行**，代表实体本身的名称。
- **H2 (`## `)**: **所有业务章节必须使用 H2**。当你使用 `append-update` 向实体写入内容时，不论你传入什么层级的标题，系统都会自动将它们**拍平 (Flatten)** 为 `##`。
- **禁止使用 H3 及以上**: 不要尝试创建更深的嵌套。

## 3. 关联模型 (Relations & Links)

知识图谱的建立依赖于两种维度的关联，这决定了你该使用哪种工具进行检索和修改。

### 3.1 显式关系 (Explicit Relations: `relation as`)
- **定义**: 记录在元数据中的强语义关联，例如 `relation as 住在家里: cafe3310`（代表实体住在 cafe3310 家）。
- **特点**: 
  - 谓词（如 `住在家里`）会被系统自动规范化。
  - 目标实体名必须是精确的实体名，不要加方括号。
  - 也有多个值的关系，如 `relation as 朋友: 张三, 李四`。
- **工具流**: 
  - 检索：使用 `search-entities --rel "住在家里:cafe3310"` 精确过滤。
  - 编辑：使用 `manage-relations` 或在 `create-entity` / `append-update` 时通过 `--add-rel-out` 附加修改。

### 3.2 隐式链接 (Implicit Links: `[[WikiLinks]]`)
- **定义**: 写在**正文**中的文本引用，例如：`昨天和 [[张三]] 开了会，定下了 [[5-1计划|5.1计划]]。`
- **特点**:
  - 使用双层方括号包裹目标实体名。支持 `[[实体名|显示文本]]` 语法，实体名是标准化的。
  - 仅仅表示“提及”，没有明确的谓词属性。
- **系统行为**: 
  - 当你执行 `rename-entity` 时，系统会自动扫描并更新全库正文中对应的 `[[WikiLinks]]`，保证不出现死链。
  - `doctor` 命令可用于排查并纠正格式不规范的链接。

## 4. 缓冲编辑范式 (Buffer-Update Editing)

建议使用 `memocli append-update` 以及 STDIN 管道模式，而不是直接覆盖文件。

- **非破坏性 (Non-destructive)**: `append-update` 不会直接修改已有正文。它会将你提供的新内容包裹在一个带有时间戳的 HTML 注释块（`UPDATE_BLOCK`）中，并追加到文件末尾。
- **避免幻觉与丢失**: 这让你可以放心地持续添加新观察到的知识，而不用担心因为上下文截断或大段重写导致原有关键信息丢失。
- **为什么用 STDIN (`echo "..." | memocli ...`)**: 因为 Markdown 内容通常包含多行、双引号、单引号和各种符号。通过命令行参数 (`-c "..."`) 传递时，极易触发 Shell 的转义错误导致命令崩溃。通过 STDIN 管道传输数据可以完美避开这个问题。
