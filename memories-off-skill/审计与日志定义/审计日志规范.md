# 审计与日志定义: Journey Specification

本文档定义了 `memory-skill` 中全局审计日志（Journey）的存储格式和顺序追加规则。

---

## 1. 存储原则 (Storage Principles)

- **文件名**: `journey.md`。
- **物理位置**: 位于知识库的根目录下。
- **写入模式**: **顺序追加 (Append-only)**。
    - 所有对知识库内容的修改动作（Mutation）都必须按发生时间顺序，在文件末尾追加记录。
    - 禁止修改或删除已记录的历史条目。

## 2. 日志条目结构 (Entry Structure)

每个日志条目采用 Markdown 列表项或表格行进行记录，以确保简洁和可读性。

### 2.1 条目字段 (Fields)
- **Timestamp**: 变更发生的时间 (`YYYY-MM-DD HH:MM:SS`)。
- **Action**: 执行的原子操作名称（如 `addContentToSection`, `renameEntity`）。
- **Target**: 受影响的实体名称（`ThingName`）。
- **Reason**: 用户/Agent 提供的变更原因。
- **Diff/Detail**: 对具体变更内容的精简描述。

### 2.2 格式示例 (Format)

我们采用 Markdown 列表结构：

```markdown
- **2024-03-28 13:00:00** | **Action**: `addContentToSection` | **Target**: `Project-X`
  - **Reason**: 记录今日技术会议达成的两项决策。
  - **Detail**: 在 ## 决策 章节追加了关于 file-ops.ts 的实现方案。

- **2024-03-28 13:05:00** | **Action**: `renameEntity` | **Target**: `Project-X` -> `Project-Core`
  - **Reason**: 项目代号已正式确认为 Core。
  - **Detail**: 重命名实体文件，并自动同步更新了 3 个 WikiLinks。
```

## 3. 写入时机 (Logging Workflow)

1.  **准备修改**: Agent 接收到修改指令。
2.  **执行修改**: 完成对实体文件、元数据或关系的物理写入。
3.  **追加记录**: 在 `journey.md` 末尾写入本次操作的条目。
4.  **完成响应**: 向用户返回 XML 报告。

## 4. 职能与用途 (Usage)

- **变更审计**: 供用户或上级 Agent 回溯知识库的演进历史。
- **恢复辅助**: 在发生误操作时，日志提供了定位回滚点的详细参考信息。
- **协作协调**: 在多 Agent 场景下，作为了解当前知识库最新动态的“公告板”。
- **Token 压缩**: 相对于每个实体一个日志，全局单一日志减少了文件发现和读取的频率，且更有利于 Agent 掌握整体变更时序。
