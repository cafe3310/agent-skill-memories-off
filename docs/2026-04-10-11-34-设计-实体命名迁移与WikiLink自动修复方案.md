# 设计文档：实体命名从 type-name.md 迁移至 name.md 的自动化方案

## 1. 背景 (Problem Statement)

用户基于近期 `memories-off` 规范的改动（实体名称由 `{entity type}-{name}.md` 变更为 `{name}.md`），需要对现有的知识库（Wiki）进行存量数据清理。

目前知识库中存在混杂的命名方式（如 `人物-阿霄.md` 和 `阿霄.md`），这导致了以下问题：
1. **文件命名不一致**：不符合最新的存储规范。
2. **WikiLinks 冲突**：部分链接可能仍指向 `[[人物-阿霄]]`，导致新规范下的检索器无法正确解析。
3. **关系引用失效**：Frontmatter 中的 `relation as ...` 字段如果包含旧的命名格式，会导致工具链在分析图谱时出现断裂。

---

## 2. 核心迁移策略 (Core Migration Strategy)

本方案采用 **“先扫描分析、后原子更名、全量修复链接”** 的逻辑。

### 2.1 识别逻辑
对于 `entities/` 目录下的每个文件 `F.md`：
1.  读取其元数据中的 `entity type`（记为 `T`）。
2.  获取不带后缀的文件名 `F`。
3.  **判断标准**：如果 `F` 以 `T + "-"` 开头，则认为其遵循旧规范。
4.  **推导新名**：`N = F[len(T)+1:]` (即移除前缀)。
    - *特例处理*：如果 `N` 为空或仅包含非法字符，则报错并跳过。
    - *冲突处理*：如果 `N.md` 已存在且不是当前文件（例如同时存在 `人物-笃挚.md` 和 `对话风格-笃挚.md`），则记录为冲突。
    - **建议方案**：对于此类冲突，自动将新名修改为 `{name}-{type_suffix}.md`（如 `笃挚-对话风格.md`），或者保持原样并列入“待人工核对清单”。本脚本将默认采取“跳过并报告”策略以确保安全。

### 2.2 更名与记录
- 执行 `git mv F.md N.md` 以保留 Git 历史。
- 维护一个全局映射表：`RenameMap = { OldName: NewName, ... }`。
- **注意**：如果发生 `笃挚-对话风格.md` 这种兜底重命名，`RenameMap` 需记录最终选定的名称。

### 2.3 全量修复 (Content Fix)
遍历库中所有 Markdown 文件（包括 `entities/*.md` 和 `meta.md`）：
1.  **修复 WikiLinks**：搜索 `[[OldName]]` 并替换为 `[[NewName]]`。
2.  **修复元数据关系**：
    - 解析 `relation as ...` 字段。
    - 如果某个目标存在于 `RenameMap` 的 Key 中，则替换为对应的 Value。
    - *注意*：需支持逗号分隔的多值解析。
3.  **更新元数据审计信息**：
    - `date modified` 更新为当前时间。
    - `reason` 设置为 `迁移: 实体命名标准化 (type-name -> name)`。

## 4. 风险控制与验证 (Risk Control)

1. **原子性**：建议先进行 `Dry Run`（模拟运行），打印出所有将要执行的更名和修改。
2. **Git 备份**：在执行前检查是否有未提交的更改，强制要求在干净的 Git 工作树上运行。
3. **回滚能力**：利用 Git 的 `git checkout .` 或 `git reset --hard` 进行回滚。
4. **验证**：
    - 运行 `memocli stats` 确认实体总数未变。
    - 随机抽取更名后的实体，检查其 WikiLinks 是否能正常跳转（使用 Obsidian 或 `memocli search`）。
