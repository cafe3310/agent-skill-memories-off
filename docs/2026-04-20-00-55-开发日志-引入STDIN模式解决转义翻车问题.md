# 开发日志: 引入 STDIN 模式以解决复杂内容的 Shell 转义问题

## 1. 背景
Agent 在使用 `memocli` 时，经常需要传输包含多行列表、Markdown 符号、各种引号、Emoji 以及 WikiLinks 的复杂字符串。在命令行模式下使用 `-c "..."` 极易因为 Shell 的转义机制导致内容损坏或命令解析失败。

## 2. 解决方案
为 `append-update` 和 `create-entity` 引入了 `--content-stdin` 模式。

### 2.1 技术实现
- **显式开关**: 增加了 `--content-stdin` 标志位。
- **互斥校验**: 脚本会强制校验 `-c/--content` 与 `--content-stdin` 不得同时出现，确保意图明确。
- **流式读取**: 当开启该模式时，脚本通过 `sys.stdin.read()` 读取完整数据流，跳过 Shell 层的字符串解析。
- **非交互安全**: 在 `run()` 方法中处理了 STDIN 读取逻辑，确保在重定向环境下能稳定获取数据。

### 2.2 文档与提示词同步
- 更新了 `操作定义/` 下的工具规格说明，明确了命令行模式与 STDIN 模式的两种执行方式。
- 更新了 `README.md`，将 STDIN 管道模式作为 Agent 的推荐实践。
- 引导 Agent 在处理长文本或包含复杂标点的内容时，优先采用 `cat/write_file` + `memocli ... --content-stdin` 的组合。

## 3. 验证情况
- 验证了 `echo "complex 'content' with [[links]]" | memocli append-update --entity "Test" --content-stdin --reason "test"` 的正确性。
- 验证了互斥校验：同时提供 `-c` 和 `--content-stdin` 时，系统正确报错。

## 4. 结论
此举大幅提升了 `memories-off` 在自动化环境下的健壮性，消除了知识库写入过程中最主要的故障源。
