# 其他 Coding Agent 的优化建议

**memocli 在“安全写入、关系管理、审计提交”这三件事上已经不错了，但在“检索定位、批量操作、实体名解析、一致性修复”这几块还不够顺手。**
这次实际浪费我时间和 token 的，主要不是“写进去”，而是“先找准、再批量改、最后校验一致性”。

下面我按“这次真实遇到的问题”列。

---

## 一、这次流程里，memocli 不顺手的地方

### 1) 实体名解析不透明，容易因为规范化吃亏（于 202604192206 已经解决）

#### 这次表现

我创建了：

- `20260411-Some-2.5-讨论`

但实际落盘变成了：

- `20260411-Some-2-5-讨论`

后面我再拿 `Some-2.5-发布` 去 `append-update` 时，直接失败了，因为工具实际上在找的是另一种规范化后的名字。

#### 带来的问题

- 会多一次失败请求
- 还要额外 `ls` / `search` / `read` 去确认真实名字
- 容易让 agent 不敢直接批量操作

#### 建议优化

做一个明确的实体解析命令，比如：

- `memocli resolve-entity "Some-2.5-更新"`
- 返回：
  - 精确匹配
  - 规范化后名字
  - 文件路径
  - aliases 命中情况

更好的做法是：

- **显示名** 和 **内部 slug** 分离
- CLI 接受显示名 / alias，内部自己 resolve

#### 优先级

**P0**

---

### 2) `create-entity` 不能直接带正文内容，导致必须两步走（于 202604192233 已经解决）

#### 这次表现

我建一个新实体，通常要：

1. `create-entity`
2. `append-update`

有时还不够，还得再：

3. `Edit` frontmatter，比如补 `aliases`

#### 这次具体例子

`张三发` 这个实体：

- 先创建
- 再 append-update 写内容
- 再用 `Edit` 手工补 `aliases: 张三`

#### 带来的问题

- 一个实体至少 2~3 次调用
- 很适合出错
- 还会让提交前状态更碎

#### 建议优化

给 `create-entity` 增加参数：

- `--content`
- `--aliases`
- `--template`
- `--frontmatter key:value`

这样很多新实体可以一次完成。

#### 优先级

**P0**

---

### 3) 缺少“按类型列实体”的官方命令，逼得我写脚本

#### 这次表现

为了找出所有 `社交媒体案例`，我直接写了 Python 扫描 frontmatter。

#### 实际上我需要的只是

- `list entities where entity type = 社交媒体案例`

#### 带来的问题

- 这类能力本来应该是 CLI 基础能力
- 结果我用了 Bash + Python，增加额外 token 和操作复杂度

#### 建议优化

加一个非常直接的命令：

- `memocli list --type "社交媒体案例"`
- `memocli list --type "工作经验" --names-only`
- `memocli list --type "人物" --with-aliases`

最好还能支持：

- `--modified-after`
- `--has-relation`
- `--missing-aliases`

#### 优先级

**P0**

---

### 4) `search --name` 有 bug（于 202604192206 已经解决）

#### 这次表现

我跑了类似这种命令后直接报错：

- `memocli search --name --names-only ...`

报的是 `UnboundLocalError`。

#### 带来的问题

- 这不是体验问题，是功能缺陷
- agent 会额外绕路改用 `--all` 或自己 grep

#### 建议优化

先把 bug 修掉。
另外建议补齐几种搜索模式：

- `--exact`
- `--name`
- `--alias`
- `--type`
- `--frontmatter`
- `--content`

#### 优先级

**P0**

---

### 5) `inspect_schema` 是采样，不适合做精确治理

#### 这次表现

它输出的是“采样自 100 个文件”，所以我后来为了确认类型分布，又自己写 Python 统计全量类型。

#### 带来的问题

- 采样对“快速感知”还行
- 但对“我刚新增了新类型，想确认全库真实状态”不够用

#### 建议优化

`inspect-schema` 增加两种模式：

- 默认：快速采样
- `--full`：全量精确扫描

输出最好区分：

- 手册定义的类型
- 实际存在的类型
- 只在手册里有、实体里没有的类型
- 实体里有、手册里没定义的类型

#### 优先级

**P1**

---

### 6) 缺少“按实体/章节加载”的 memocli 命令，导致我退回 Read 工具

#### 这次表现

虽然 skill 文档强调应该用自己的检索/加载能力，但实际 CLI 可见命令里，没有一个特别顺手的：

- 按实体名加载 frontmatter
- 只加载某个 section
- 只加载摘要 / TOC

所以我大量用了：

- `Read /absolute/path/to/file`

#### 带来的问题

- agent 需要知道真实文件路径
- 会读进比需要更多的内容
- token 浪费明显

#### 建议优化

加这类命令：

- `memocli load --entity "张三" --frontmatter-only`
- `memocli load --entity "张三" --toc`
- `memocli load --entity "张三" --section "个人简况"`
- `memocli load --entity "张三" --summary`

#### 优先级

**P0**

---

### 7) 批量改动没有事务式 batch，导致我只能写超长 Bash 串

#### 这次表现

我有一轮是连续：

- create entity
- append update
- append another
- update relations
- update multiple existing entities

最后变成一大串 Bash `&&`。

#### 带来的问题

- shell quoting 很脆
- 中间一条失败，后面全断
- 很难复用
- 很难预览变更

#### 建议优化

做一个 batch 模式：

- `memocli apply-batch plan.yaml`

例如：

```yaml
operations:
  - create_entity:
      name: 张三发
      type: 人物
      aliases: [张三]
      content: ...
  - create_entity:
      name: 20260411-Some-2-5-发布讨论
      type: 对话记录
      relations:
        out:
          参与: [张四, 王五, 赵六]
  - append_update:
      entity: Some-2-5-发布
      content: ...
reason: 理解 2026-04-11 群聊...
```

#### 优先级

**P0**

---

### 8) 长文本内容通过 shell 参数传递，很难受

#### 这次表现

`append-update -c $'...'` 这种写法很长，里面又有：

- 引号
- emoji
- 中文标点
- 多行列表
- wiki links

很容易因为转义翻车。

#### 带来的问题

- 影响稳定性
- 也影响可读性
- 复杂内容越多越难维护

#### 建议优化

支持：

- `--content-file xxx.md`
- `--content-stdin`

比如：

```bash
cat payload.md | memocli append-update --entity "王五" --content-stdin --reason "..."
```

#### 优先级

**P0**

---

### 9) `rename-type` 的“同步 meta”能力不完整

#### 这次表现

`rename-type` 把实体类型迁好了，但 `meta.md` 里关系约定那句：

- `... / 社交媒体案例`

并没有一起改掉。

#### 带来的问题

- 工具说“同步 meta.md”，但实际只是同步了一部分
- 还要我再补 `Edit`

#### 建议优化

`rename-type` 应该支持两档：

- 安全模式：只改 frontmatter 中的类型定义
- 全同步模式：连 `meta.md` 里的说明性文本一起改，并给出 diff 预览

最好加：

- `--update-manual-references`

#### 优先级

**P1**

---

### 10) 提交语义不统一

#### 这次表现

大部分命令是先改、最后再 `memocli commit`。
但 `rename-type` 这次直接自己提交了 Git 变更。

#### 带来的问题

- agent 很难预测哪些命令会自动 commit
- 容易把一次逻辑相关的修改拆成多个 commit
- 不利于整理审计历史

#### 建议优化

统一策略，二选一：

#### 方案 A

所有 mutating 命令都只改工作区，不自动 commit

#### 方案 B

所有 mutating 命令都要求 `--reason`，并自动 commit

我更建议 **方案 A**，再保留一个显式 `commit`。

#### 优先级

**P1**

---

### 11) 缺少“一致性校验”命令（于 202604192206 已经解决）

#### 这次表现

我为了确认有没有旧类型残留，又手工跑了：

- `rg`
- `git diff`
- `inspect_schema`

#### 建议优化

做一个：

- `memocli validate`

检查内容包括：

- meta 定义的类型 vs 实际类型
- 是否有未定义类型
- 是否有手册残留旧类型名
- WikiLinks 是否悬空
- relation target 是否不存在
- alias 是否冲突

#### 优先级

**P0**

## 二、哪些地方明显浪费了 token 和请求轮次

下面这些地方，是这次最值得优化的。

### 1) 反复“搜索 -> ls -> Read -> 再修正名字”

#### 这次典型浪费

因为实体名规范化不透明，我多走了：

- 搜索
- ls
- 读取文件
- 再重试 append

#### 优化方式

加 `resolve-entity`

#### 节约效果

能直接少掉一整轮失败和后续确认，**非常值**。

---

### 2) 为了查类型分布和旧类型实体，我写了 Python 脚本

#### 这次典型浪费

我写脚本做：

- 列出所有 `社交媒体案例`
- 全量统计 entity type

#### 优化方式

加：

- `list --type`
- `inspect-schema --full`

#### 节约效果

能把 2~3 个 tool call 压成 1 个，还省掉脚本输出的 token。

---

### 3) 读整文件太重

#### 这次典型浪费

我很多时候只是想看：

- frontmatter
- 某个 section
- 当前关系
- 最近 update block

但因为没好用的 memocli load，只能整篇 `Read`。

#### 优化方式

加：

- `load --frontmatter-only`
- `load --toc`
- `load --section`
- `load --recent-updates 3`

#### 节约效果

这块是 **最实打实省 token** 的地方。
很多读取理论上能压到原来的 **20%~40%**。

---

### 4) XML 输出太长

#### 这次典型浪费

每次结果都有：

- `<memocli-result ...>`
- `<source-sh>...`
- `<content>...`

对于 agent 来说很稳，但也确实长。

#### 建议优化

给一个简洁模式：

- `--quiet`
- `--json`
- `--no-source-sh`

例如只返回：

```json
{
  "ok": true,
  "touched_entities": ["赵六", "技术方案"],
  "created": ["评论驱动的方案设计经验"]
}
```

#### 节约效果

如果高频调用，这个能明显降 token。

---

### 5) 一次逻辑任务拆成太多命令

#### 这次典型浪费

“建立新类型 + 新建领域桶 + 从对话抽 3 条经验 + 关联回项目”
本来是一个认知动作，但实际拆成很多命令。

#### 优化方式

做 batch / transaction：

- 一次传入操作计划
- 一次返回执行结果
- 一次 commit

#### 节约效果

这种复杂编辑任务，粗略能把 **10+ 次调用压到 1~3 次**。

---

### 6) 没有“上下文打包”命令

#### 我这次理想上想要的是

比如我输入：

- “帮我了解工程设计经验相关上下文”

工具应该能一条命令给我：

- 相关实体列表
- 每个实体的 3~5 行摘要
- 最近更新
- 可能的关系图入口

#### 建议优化

增加类似：

- `memocli context-pack --topic "工程设计经验"`
- `memocli context-pack --entities "赵四,王五,Jira"`

#### 节约效果

能大幅减少：

- search
- read
- open
- 再 search

这种往返轮次。

---

## 三、如果只做少数几个优化，我建议优先做这些

### P0：最值得马上做

1. `resolve-entity`
2. `create-entity` 支持 `--content` / `--aliases`
3. `load` 系列命令：frontmatter / section / toc / summary
4. `list --type`
5. `validate`
6. `append-update` 支持 `--content-file` / `--content-stdin`
7. `batch apply`

---

### P1：第二优先级

1. 修复 `search --name` bug
2. `inspect-schema --full`
3. 统一 commit 语义
4. `rename-type` 完整同步手册说明文本
5. 精简输出模式 `--quiet` / `--json`

## 四、我个人觉得已经做得不错的地方

也说句公道话，不是全都不顺手。

现在 memocli 已经比较好的部分：

- 在知识库根目录自动注入 path，这个很省心
- `append-update` 的非破坏式编辑思路是对的
- relation 管理整体方向对
- commit 带 reason 的审计机制很适合长期知识库

所以它的问题不是“方向错了”，而是：

**已经进入“可用”阶段，但还没进入“非常适合 agent 高密度作业”的阶段。**

---

## 五、如果你要我给一个最实用的产品化建议

我会建议直接补一个：

### `memocli apply-plan`

输入一份结构化计划，内部自动做：

- resolve entity
- create / append / relations
- validate
- stage changes
- optional commit

这会把现在 agent 侧大量 shell 编排的工作，直接吃回工具里。
**这是最能同时节约 token、轮次、失败率、以及 prompt 复杂度的一步。**

如果你愿意，我下一条可以直接把这些建议整理成一份更工程化的清单，按：

- 问题
- 复现场景
- 建议命令设计
- 优先级
- 预期收益

输出成适合你拿去开 Jira 的版本。
