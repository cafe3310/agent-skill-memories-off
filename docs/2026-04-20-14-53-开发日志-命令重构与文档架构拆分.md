# 开发日志: CLI 语义化重构与 Agent 文档架构精简

## 1. 背景与目标
随着 `memories-off` 技能所支持的命令（CLI 工具）数量增加，现有的扁平化帮助菜单显得杂乱无章，增加了用户（特别是 Agent）在初次建立上下文时的认知成本。此外，原有的 `SKILL.md` 和各分项规格文档事无巨细地罗列了所有的设计细节和底层逻辑，导致输入给 Agent 的 Token 负担极高，且模糊了“如何使用工具”与“工具如何被构建”的边界。

本次优化的目标是：
1. 对现存命令进行语义化重命名与视觉分组。
2. 彻底拆分技能文档，区分面向 Agent 的“使用手册”和面向开发者的“设计规范”。

## 2. 核心变更内容

### 2.1 CLI 语义化重命名与规则总结
为了使命令更加符合操作直觉和业界惯例，我们对多项底层操作进行了重命名。总结出的命名经验（最佳实践）为：
- **强调目标规模 (单复数分离)**：一次仅能操作单体对象时使用单数，如 `rename-entity`, `create-entity`；一次可能操作集合、返回列表时使用复数，如 `search-entities`, `load-entities`, `merge-entities`。
- **动词借喻强化认知**：将晦涩的 `consolidate-updates` 变更为 `squash-updates`，借用 Git 中 `squash` 的比喻，生动表达“压缩并合并碎片”的操作本质，大幅降低 Agent 的理解门槛。

### 2.2 CLI 帮助信息分组呈现
在底层 `ScriptBase` 引入了 `group_name` 属性，并在 `install.py` 中实现了根据预设权重对子命令进行分类展示的逻辑。如今 `memocli --help` 按照用户工作流的先后次序展示：
1. `【核心探索 (Core Exploration)】`
2. `【知识写入 (Knowledge Writing)】`
3. `【检索与加载 (Search & Loading)】`
4. `【结构化编辑 (Structured Editing)】`
5. `【系统与维护 (System & Maintenance)】`

### 2.3 文档架构深度拆分
这是本次优化中针对 LLM 认知负荷最重要的改造：
- **底层剥离**：新建 `design-doc/` 目录，将原本直接暴露给 Agent 的 `操作定义/`、`文档结构定义/`、`关联定义/` 等“造轮子指南”全部封存。明确声明这些文档仅用于开发维护。
- **重塑 SKILL.md**：主文档不再事无巨细地重复 CLI `help` 中的参数说明，而是聚焦于：**Agent 应该遵循什么工作流（先 explore，后看 help），以及必须遵守什么死板规则（如必须用 append-update，强制 STDIN 传参，H1/H2 约束）**。
- **精炼 Reference**：在 `references/` 下新增了为 Agent 量身打造的快速认知材料：
  - `core_concepts.md`: 精简科普诸如 Slug, Frontmatter, WikiLinks 隐式双链等核心领域模型。
  - `batch_operations.md`: 指导 Agent 如何节约 Token 和减少调用轮次（如 `--names-only`、一次性注入关联、Bash `for` 循环），这显著提升了它在自动模式下的聪明程度。

## 3. 经验与结论
- **帮助信息系统化远胜于文档重复**：现代 Coding Agent 有极强的使用 CLI `--help` 的能力。将使用说明收敛于工具的动态 `--help` 并在主控 Prompt/SKILL.md 中加以指引，比在 Markdown 里维护静态参数表要安全、高效得多。
- **隔离“怎么用”与“怎么做”**：Agent 作为工具的“使用者”，不需要也不应该去通读工具底层的物理存放规范。精简参考资料、提炼核心模型（Core Concepts）并在必要时传授技巧（Batch Operations），能够完美平衡安全性与上下文压力。
