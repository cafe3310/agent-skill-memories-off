# 需求与设计: 实体章节 (Header) 结构标准化

## 1. 背景与目标
在长期维护的知识库中，用户或 Agent 通过 `append-update` 或者直接创建 (`create-entity`) 时，经常随意使用不同层级的 Markdown 标题（例如 `#`, `###`, `####` 等）。这使得：
1. 实体内部结构混乱，缺乏统一的 `TOC` (目录) 层级。
2. 难以基于标准的结构实现“加载特定章节”的工具（这也是后续优化 `load` 命令的前置条件）。

为了实现未来“按章节检索和列出实体内容”的高级能力，我们需要引入一套机制：**所有的实体中，除了最开头的 H1 (`# 实体名`) 外，所有其他业务内容的 Header 均统一标准化为 H2 (`## 章节名`)**。如果遇到深层级的子标题，在当前扁平化的知识库结构要求下，也会被“拍平”为 H2，或者统一规范到一个严格的 H2 层级。

## 2. 方案设计

### 2.1 底层抽象：`MetadataParser.normalize_headers`
在 `utility/schema_define.py` 的 `MetadataParser` 中添加一个静态方法 `normalize_headers(text: str) -> str`。
- **匹配规则**: `^#+\s+(.*)` （支持多行匹配，匹配所有以至少一个 `#` 开头的行）。
- **处理规则**: 将其替换为 `## \1`（即强制转换为 H2）。
- **例外处理**: 需要注意，如果在正文中有代码块 (```) 内含有 `#`，标准的正则处理可能会误伤。为了安全，最简单的实现是按行遍历，当不在代码块内时，遇到以 `#` 开头的行则执行替换。

### 2.2 拦截写入：`UpdateBlockManager.create_block`
正如我们之前拦截 WikiLink 一样，既然所有的动态写入现在都已经收拢到了 `UpdateBlockManager.create_block(content, reason)` 中，我们只需要在 `create_block` 内部新增调用：
```python
content = MetadataParser.normalize_headers(content)
```
这样，无论是 `create-entity --content` 还是 `append-update --content` 带来的增量信息，都会被自动转为 H2。

### 2.3 `doctor` 存量诊断与修复
在 `doctor.py` 中：
- 如果 `normalize-name` （或者单独创建一个 `--normalize-headers` / 整合进 `--audit`）被执行，应当扫描实体文件的 `body`（去除了最前面的 `# EntityName` 之后的纯正文）。
- 使用 `normalize_headers` 对 `body` 进行处理，检查是否有变化。
- 若有，则提示并修复。

由于现在的实体结构是：
```markdown
---
[Frontmatter]
---

# {EntityName}

{Body with UPDATE_BLOCKS}
```
我们扫描时，可以先提取第一行 `# {EntityName}` 予以保留，然后对剩余的内容执行 `normalize_headers`。

## 3. 行动步骤
1. 在 `schema_define.py` 中实现安全的 `normalize_headers`（忽略代码块内部）。
2. 在 `schema_define.py` 的 `create_block` 中调用该方法，保障所有新产生的 `UPDATE_BLOCK` 均被强制 H2 化。
3. 在 `doctor.py` 的 `_run_normalize_name_pass` 或新 pass 中，增加对正文头部的全局清洗，保留顶部的 `# {EntityName}`。
4. 执行测试验证。
