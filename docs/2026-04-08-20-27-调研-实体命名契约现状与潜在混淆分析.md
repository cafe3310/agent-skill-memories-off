# 调研报告：实体命名契约现状与潜在混淆分析 (Naming-Confusion-Analysis)

## 1. 现状调研 (Status Quo)

### 1.1 `create-entity` 的实现逻辑
通过分析 `memories-off-skill/scripts/create_entity.py`：
- **输入**: `--name (-n)` 和 `--type (-t)`。
- **文件名生成**: `file_path = ctx.entities_path / f"{normalized_name}.md"`。
- **事实**: 该工具**不会**自动将 `--type` 拼接到 `--name` 前面。
- **后果**: 如果用户输入 `--name "b" --type "a"`，生成的文件是 `b.md`，而非 `a-b.md`。

### 1.2 `meta.md` (库手册) 的建议规范
在 `init.py` 生成的 `meta.md` 模板中提到：
> "实体文件名：使用 [类型]-[名称] 格式。"
- **事实**: 这是一个**软性建议**（惯例），并未在代码中强制执行或自动补全。

### 1.3 `append-update` 等后续工具的逻辑
- **输入**: `--entity`。
- **查找**: `target_file = ctx.entities_path / f"{normalized_name}.md"`。
- **事实**: 工具只寻找用户输入的物理文件名。

---

## 2. 潜在混淆点 (Potential Confusions)

### 混淆 A：Agent 的认知偏差
如果 Agent 在 `meta.md` 看到“建议使用 [类型]-[名称]”，它可能会有两种行为：
1. **主动拼接**: 调用 `create-entity --name "a-b" --type "a"`。此时文件名是 `a-b.md`。
2. **直觉调用**: 调用 `create-entity --name "b" --type "a"`。此时文件名是 `b.md`。

**后果**: 
当 Agent 稍后想 `append-update` 时，它可能不确定该找 `b` 还是 `a-b`。如果它使用了与创建时不同的习惯，就会报错“实体不存在”。

### 混淆 B：冗余存储
如果 Agent 主动拼接了（行为 1），那么文件的 Frontmatter 会变成：
```yaml
entity type: a
...
```
而文件名是 `a-b.md`。此时“类型”信息在路径和内容中发生了冗余。如果将来用户想把实体 `b` 的类型从 `a` 改为 `c`，则必须同时执行 `rename` (a-b -> c-b) 和更新元数据。

---

## 3. 改进方向建议 (Proposed Directions)

为了消除这种“由谁来拼前缀”的暧昧性，我们有以下两种选择：

### 方案 1：强制自动拼接 (Strict Enforced Naming)
- **逻辑**: `create-entity` 接收 `-t a -n b`，强制生成 `a-b.md`。
- **读取**: `append-update --entity b` 时，要求工具能自动在 `entities/` 下寻找 `*-b.md`（可能冲突）或要求用户必须输入完整名。
- **缺点**: 增加了查找逻辑的复杂性，且破坏了“物理名即 ID”的简洁性。

### 方案 2：显式解耦与文档修正 (Explicit Decoupling)
- **逻辑**: 明确规定“文件名即实体名”，**不再建议**在文件名中包含类型。
- **修改**: 将 `meta.md` 的建议改为“文件名建议简洁唯一，类型信息由元数据维护”。
- **优点**: 极其简洁，不存在拼接逻辑的黑盒。
- **缺点**: 在文件系统中浏览时，无法直观看出实体类型。

---

## 4. 结论与请示 (Conclusion)

目前的现状是：**工具链是解耦的，但文档建议是耦合的**。

**我的看法**: 
建议采用**方案 1 的变体（智能助手化）**：
1. 在 `create-entity` 中，如果用户输入的 `--name` 不包含类型前缀，则根据 `--type` **自动补全**前缀。
2. 在所有涉及读取的工具中，明确文档提示：`--entity` 必须提供完整的物理文件名（含前缀）。
