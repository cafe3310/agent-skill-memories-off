# 2026-03-29-17-32-设计-缓冲更新与梦境整理机制 (Buffer-then-Dreaming)

## 1. 背景与问题 (Background)
在大语言模型 (LLM) 更新现有 Markdown 实体时，传统的“原地编辑”或“全量重写”模式存在极高的风险：
- **内容丢失**: LLM 可能在处理超长文本时忽略掉原有段落。
- **定位失败**: 基于字符串匹配或行号的编辑工具极易因为 Markdown 格式微调而失效。
- **审计困难**: 除非人工逐行对比，否则很难一眼看出更新过程中是否意外删除了信息。

## 2. 核心方案：缓冲更新 (Buffer-then-Dreaming)
本方案参考 Anthropic 的 "Dreaming" 思路，将**正文内容**的更新行为拆分为“异步追加”和“同步整理”两个阶段。注意：本机制不适用于 Frontmatter (YAML) 区域，元数据变更应使用专用管理脚本。

### 2.1 阶段一：更新追加 (Append-only Buffer)
Agent 不再尝试直接修改文件中间的内容，而是将所有的**章节更新意图**以一个标准的“更新块 (Update Block)”形式追加到实体文件的末尾。


**更新块格式规范**:
```markdown
--- [UPDATE BLOCK] ---
timestamp: YYYY-MM-DD HH:MM:SS
target_heading: "## 目标标题" (若为新章节可为空)
action: "append" | "replace" | "new_section"
content: |
  要更新或追加的具体 Markdown 内容...
--- [END OF UPDATE BLOCK] ---
```

### 2.2 阶段二：梦境整理 (Dreaming / Consolidation)
这是一个由用户手动触发或在特定时机运行的自动化脚本。它负责解析文件末尾的所有更新块，并将它们“消化”到正文的正确位置中。

**处理逻辑**:
1.  **扫描**: 遍历所有实体，识别包含 `--- [UPDATE BLOCK] ---` 的文件。
2.  **解析**: 提取每一个更新块的 `target_heading`, `action` 和 `content`。
3.  **合并**:
    - `append`: 在目标标题对应章节的末尾追加内容。
    - `replace`: 替换目标标题对应章节的所有正文。
    - `new_section`: 在文件末尾（最后一个更新块之前）插入新章节。
4.  **清理**: 删除已处理的更新块标记。
5.  **审计**: 自动执行 `git commit`，记录本次“梦境整理”所影响的所有实体。

## 3. 实施计划 (Implementation Plan)

### 第一阶段：脚本开发
1.  **`append_update.py`**:
    - 职能: 接受参数并向指定实体末尾追加格式化的更新块。
    - 参数: `--path`, `--entity`, `--target_heading`, `--action`, `--content`。
2.  **`consolidate_updates.py`**:
    - 职能: 核心算法，执行“梦境整理”逻辑。
    - 特性: 必须具备高度的容错性，确保在合并失败时保留原始更新块不丢失。

### 第二阶段：规范与引导更新
1.  **更新 `SKILL.md`**: 明确将“缓冲更新”作为 Agent 更新实体的**首选推荐方式**。
2.  **更新 `编辑工具规格.md`**: 
    - 标注 `replaceSection` 等工具为“低优先级/高风险”。
    - 新增 `appendUpdateBlock` 工具描述，并关联 `append_update.py`。

### 第三阶段：集成测试
1.  在模板知识库上模拟 Agent 的多次追加更新。
2.  手动触发 `consolidate_updates.py`，观察合并后的内容完整性。

## 4. 优势 (Benefits)
- **绝对安全**: 追加操作永远不会破坏原有数据。
- **审计友好**: 用户可以在整理前预览文件末尾的所有“待处理”更新。
- **降低延迟**: 追加操作比复杂的解析+重写逻辑更快，Token 消耗更低。
