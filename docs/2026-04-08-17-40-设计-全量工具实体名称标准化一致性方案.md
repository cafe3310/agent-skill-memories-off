# 设计文档：全量工具实体名称标准化一致性方案 (Entity-Naming-Consistency)

## 1. 背景 (Problem Statement)

在 `memories-off` 规范中，实体在文件系统中的存储名称是经过标准化的（如 `A:B` 存储为 `A-B.md`）。目前的工具集中，部分工具（如 `create-entity`）在写入前执行了标准化，但另一部分工具（如 `get-relations`, `append-update`）在读取或指定目标时直接使用了用户输入的原始字符串。

**后果**:
- 用户输入 `memocli get-relations --entity "人物:张三"`，工具会去寻找 `人物:张三.md`（报错不存在），而实际文件是 `人物-张三.md`。
- Agent 在多个工具间链式调用时，会因为这种不一致性导致逻辑中断。

---

## 2. 核心原则 (Core Principle)

**“输入宽容，存储严格”**：所有子命令在接收到涉及“实体名称”的参数时，必须在第一时间调用 `MetadataParser.normalize_name()` 进行预处理。

---

## 3. 实施方案 (Implementation Plan)

### 3.1 权威逻辑确认
统一使用 `memories-off-skill/scripts/schema_define.py` 中的 `MetadataParser.normalize_name()`。
- **当前规则**: 匹配所有非字母、非数字、非中文的字符，并替换为单中划线 `-`。

### 3.2 待修正工具清单
| 工具 | 参数 | 修正动作 |
| :--- | :--- | :--- |
| `append-update` | `--entity` | 对输入名称执行标准化。 |
| `get-relations` | `--entity` | 对输入名称执行标准化。 |
| `manage-relations` | `--source`, `Target (in --add/--remove)` | 对源和目标实体名执行标准化。 |
| `merge-entities` | `--sources`, `--target` | 对所有列表项及目标执行标准化。 |
| `search` | `pattern` (仅当 `--name` 时) | 如果是搜索名称，自动对模式进行标准化尝试。 |
| `rename` | `--old`, `--new` | (已部分实现) 确保 `old` 也被标准化以增强查找鲁棒性。 |

### 3.3 搜索工具的特殊处理 (`search.py`)
- **场景 A (正文搜索)**: 不进行标准化。用户搜 `A:B` 应该搜到原始文本。
- **场景 B (名称搜索 `--name`)**: 
    - 如果用户搜 `A:B`，我们应该尝试匹配物理文件名 `A-B.md`。
    - **改进**: 在执行 `ls | grep` 前，先将 `pattern` 按照实体名规则标准化。

---

## 4. 预期收益 (Benefits)

1. **消除断裂**: Agent 可以自由地在参数中使用各种符号（如 `:`, `/`, ` `），工具链会自动将其映射到物理存储。
2. **容错性**: 提升了人类用户在 CLI 下的操作体验，不再需要手动记忆文件系统的转义结果。
3. **搜索精准**: 解决了“明明有这个文件（名称包含符号），但搜不到”的尴尬。

---

## 5. 验证标准 (Validation)

1. 运行 `memocli create-entity -n "测试:名称" -t "测试"` -> 生成 `测试-名称.md`。
2. 运行 `memocli get-relations --entity "测试:名称"` -> 应该能正确找到并分析该文件。
3. 运行 `memocli search "测试:名称" --name` -> 应该能匹配到该实体。
