# 开发日志: Search 命令全面强化与纯 Python 检索重构

## 1. 背景与问题
在知识库管理中，搜索是使用频率最高的基础工具之一。但在过去的实现中，`search` 命令存在以下体验和功能瓶颈：
- 强依赖子进程调用 `grep`，导致难以精细化解析 Markdown Frontmatter。
- 搜索实体名称（`--name`）时仅搜索文件路径，无法匹配知识库内记录的别名 (`aliases`)。
- 缺少精准的分类和过滤手段，例如按类型查找 (`list --type`) 或按关系查找，使得 Agent 常常需要手写脚本来完成本应属于基建范围的查询。

## 2. 主要修改内容

为彻底解决这些问题，我对 `memories-off-skill/scripts/commands/search.py` 进行了深度重构。

### 2.1 底层逻辑：从 Bash Grep 到 Pure Python
- **抛弃子进程**：移除了所有的 `subprocess.run(grep)` 逻辑，将整个扫描过程迁移至 Python 原生实现。
- **全解析扫描**：现在每次搜索都会规范化地通过 `MetadataParser.split_content` 读取实体的元数据与正文，这为后续的精确过滤打下了数据基础。

### 2.2 功能扩展：引入属性过滤器
新增了针对元数据的专门过滤参数，使其不仅具备文本检索能力，也能作为“实体筛选列表”使用：
- **`--type` (类型过滤)**: 支持使用正则表达式过滤 `entity type` 字段。例如 `memocli search --type "人物|组织" --names-only` 即可充当缺失的 `list --type` 功能。
- **`--rel` (关系过滤)**: 支持按声明的关系过滤。格式强制为 `谓词模式:目标模式`（如 `friend:张三`）。搜索时会自动提取 `relation as ...` 字段，并在两端分别应用正则表达式匹配，精确抓取声明了对应关系的实体。
- **参数解耦**：允许主搜索 `pattern` 为空，意味着用户可以纯粹使用 `-t` 或 `--rel` 来筛选实体列表（变相实现了 List 功能）。

### 2.3 检索体验增强
- **别名感知 (`--name`)**: 当在 `--name` 或 `--all` 范围内进行检索时，引擎不仅匹配文件名，还会自动提取该实体的 `aliases` 并一并校验。只需一个名字或小名，就能直接锁定本体。
- **废弃 `--meta`**: 由于引入了极度精确的 `--type` 和 `--rel` 过滤器，原有意义不明且范围宽泛的 `--meta` 搜索被正式废弃。
- **自研 Context 引擎**: 为了在抛弃 `grep` 的同时保留 `--context (-C)` 打印上下文行数的功能，通过区间合并算法（Interval Merge）手写实现了平滑的上下文窗口抽取逻辑。

## 3. 验证与总结
所有修改均在临时初始化的 `test_kb` 中通过了严格的集成测试：
1. 成功根据别名 `"老张"` 匹配到本体 `"张三.md"`。
2. 成功通过 `--type "人物" --names-only` 列出了仅有的人物实体列表。
3. 成功通过 `--rel "friend:.*张"` 正则筛选定位到了明确定义了双向关系的对应实体。
