# 2026-03-29-17-45-设计-结构化元数据与关系管理脚本

## 1. 概述 (Overview)
为了保证知识库中结构化元数据（Frontmatter）的严谨性，特别是实体类型（Type）和语义关系（Relation）的准确，本项目需要一套专用的管理脚本。这些操作具有库级别的全局影响，不适用于异步的缓冲区机制，必须即时执行并审计。

## 2. 脚本清单 (Script Inventory)

### 2.1 `rename_type.py` (新工具)
- **职能**: 在全库范围内重命名一个实体类型。
- **逻辑**: 
    1. 扫描 `entities/` 目录下所有实体。
    2. 若实体的 `entity type` 字段匹配 `old_type`，则更新为 `new_type`。
    3. 同步更新 `meta.md` 中的类型定义。
    4. 自动执行 Git Commit。
- **参数**: `--old`, `--new`, `--reason`。

### 2.2 `merge_types.py` (新工具)
- **职能**: 将一个或多个源类型合并到目标类型中。
- **逻辑**:
    1. 接受源类型列表。
    2. 遍历库内实体，将符合源类型的实体全部更新为目标类型。
    3. 调用 `rename_type.py` 的逻辑同步 `meta.md`。
    4. 自动执行 Git Commit。
- **参数**: `--sources`, `--target`, `--reason`。

### 2.3 `manage_relations.py` (工具升级)
- **职能**: 替代原有的手动编辑，通过脚本精确建立或删除语义关系。
- **模式**:
    - `add`: 向实体的 Frontmatter 添加一个 `relation as [predicate]: [target]`。
    - `delete`: 移除指定的关系项。
- **特性**: 
    - 自动校验目标实体是否存在（Orphan check）。
    - 自动更新实体的 `date modified`。
    - 自动调用 `commit.py`。
- **参数**: `--entity`, `--action (add|delete)`, `--predicate`, `--target`, `--reason`。

## 3. 实施计划 (Implementation Plan)
1. **优先实现 `manage_relations.py`**: 因为这是最频繁的原子操作。
2. **随后实现 `rename_type.py` 和 `merge_types.py`**: 用于库级别的结构重整。
3. **更新 `SKILL.md` 与工具规格**: 将标注从“手动/Agent行为”更新为“脚本驱动”。

## 4. 优势 (Benefits)
- **零误差**: 避免了 Agent 在手动修改 YAML 时可能导致的语法错误。
- **强一致性**: 确保了 `meta.md` 与实际数据类型的同步。
- **完整审计**: 每一次结构性变动都有明确的 Git 记录。
