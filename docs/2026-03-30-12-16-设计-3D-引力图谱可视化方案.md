# 设计方案: 3D 引力知识图谱可视化 (3D-Gravity-KG)

## 1. 概述 (Overview)

基于用户希望为 `memories-off` 知识库添加「3D 引力知识图谱」展示的需求，本方案旨在构建一个基于「引力与斥力平衡」的 3D 交互式知识图谱。

本方案的核心目标是**功能完备且极低干扰**：通过自包含的 HTML 渲染技术和隔离的文档策略，确保该工具在不被使用时不会占据 Agent 的上下文注意力。

### 1.1 核心设计愿景
- **语义重力化**: 相同类型的实体天然吸引，不同类型的实体相互排斥。
- **层级紧凑化**: 别名（Alias）作为实体的卫星节点，与其主实体保持极高的绑定强度。
- **低干扰集成**: 仅在需要时通过 CLI 唤起，不污染核心 Skill 定义。

---

## 2. 核心功能需求 (Functional Requirements)

### 2.1 节点与关系 (Nodes & Links)
- **实体节点**: 呈现为核心球体，标签仅显示实体名（不含类型）。
- **别名节点**: 呈现为小型卫星球体，紧密绑定于主实体周围。
- **类型引力**: 相同类型的节点通过物理引擎的自定义力场实现聚类。
- **关系连线**: 实体间的逻辑关联（显式 relations 与隐式 WikiLinks）通过线缆连接。

### 2.2 渲染策略 (Self-contained Rendering)
- **零服务端**: 脚本生成一个包含所有逻辑和数据的单文件 HTML。
- **CDN 依赖**: 引入 UMD 版本的 `react-force-graph-3d` (通过 unpkg.com)，无需本地构建环境。
- **离线数据**: 知识库数据在导出阶段被转化为 JSON 字符串并直接硬编码（Inline）到 HTML 的 `<script>` 标签中。

---

## 3. 技术架构 (System Architecture)

### 3.1 核心脚本: `scripts/force_graph.py`
这是整个可视化功能的唯一入口，负责以下任务：
- **数据解析**: 扫描整个知识库，提取实体、别名、类型及双向关系。
- **模板填充**: 将解析后的 JSON 数据注入到预定义的 HTML 渲染模板中。
- **临时分发**: 在系统临时目录（如 `/tmp` 或 `AppData/Local/Temp`）生成 `.html` 文件。
- **自动唤起**: 使用 Python 的 `webbrowser` 模块自动开启系统默认浏览器进行展示。
- **容错处理**: 若自动开启失败，向终端输出文件绝对路径供手动访问。

### 3.2 隔离文档策略 (Documentation Isolation)
为了避免 Agent 在执行日常任务时因「发现可视化工具」而分心（Attention Loss），采取以下策略：
- **核心定义隐藏**: `SKILL.md` 中**禁止**提及任何关于可视化或图谱的描述。
- **专项规格定义**: 创建 `memories-off-skill/操作定义/图谱可视化规格.md`，仅在 Agent 需要研究如何绘图时由其主动查阅。
- **CLI 最小化提示**: `memocli graph` 的 `--memo-cli-info` 仅保留一句话描述（如：“生成并开启 3D 知识图谱展示”），确保在 `memocli --help` 中不占据过多 Token 权重。

---

## 4. 详细物理场实现 (Physics implementation)

在 HTML 模板内置的 JavaScript 代码中，通过 `react-force-graph` 暴露的 D3 接口实现：

1.  **别名绑定**: 为 `ALIAS_OF` 类型的 Link 设置极短的 `distance` 和极高的 `strength`。
2.  **类型引力 (Type-Centric Force)**:
    - 预计算每种 `type` 的实时质心。
    - 在每一帧迭代中，为同类型节点施加一个指向其类型质心的向心力。
3.  **视觉区隔**:
    - `nodeAutoColorBy`: 映射至 `type`。
    - `nodeVal`: 实体节点 val=10，别名节点 val=3。
