# 计划: 精简 stderr 与全量 XML 错误报告方案 (Streamlined-Error-Flow)

## 1. 现状与冲突 (Current Conflict)

在目前的 `memocli` 错误处理中，我们面临两个矛盾的需求：
1. **Agent 友好 (Agent-First)**: 希望所有错误、参数缺失、用法说明都封装在 XML 中，方便 Agent 解析和纠错。
2. **标准兼容 (Standard-Compliant)**: 担心完全静默 `stderr` 会导致传统 Shell 脚本或 CI 环境无法感知错误（通常它们通过 `stderr` 是否为空来判断）。
3. **消除冗余 (No Duplication)**: 用户不希望在控制台看到两次相同的错误描述（一次在 XML 外，一次在 XML 内）。

---

## 2. 优化计划: “信号与上下文分离”

我们决定将 `stderr` 作为**“报错信号”**，将 `stdout` (XML) 作为**“报错上下文”**。

### 2.1 stderr: 极简信号
`stderr` 不再输出详尽的 `usage` 或具体的参数错误列表。它只输出一行极简的、符合传统的报错信息。
- **输出内容**: `memocli: error: Command failed. See stdout (XML) for details.`
- **作用**: 
    - 满足 Shell/IDE 对 `stderr` 的监控。
    - 触发非零退出码（`exit 2`）。
    - 告诉人类用户去哪里看详情。

### 2.2 stdout (XML): 全量上下文
所有的纠错逻辑（具体缺了哪个参数、该指令的详细 `help` 文档）全部收拢到 `<memocli-result>` 的 `<error-detail>` 中。
- **输出内容**: 包含原始错误消息 (`message`) + `print_help()` 的全量输出。
- **作用**: 
    - 为 Agent 提供一站式的“失败原因 + 修正参考”。
    - 确保 Transcript（对话记录）中只有一份结构化的错误说明。

---

## 3. 实施细节 (Implementation)

在 `CustomArgumentParser.error` 中：

```python
def error(self, message):
    # 1. 信号: 输出到 stderr (极简)
    sys.stderr.write(f"error: {message}\n(See XML output for full help and usage)\n")

    # 2. 上下文: 输出到 stdout (全量 XML)
    print(f'<memocli-result ... result="failed">')
    # ...
    print(f"  <error-detail>")
    print(f"错误原因: {message}\n")
    self.print_help() # 这里输出到 stdout
    print("  </error-detail>")
    print("</memocli-result>")
    
    sys.exit(2)
```

---

## 4. 预期效果 (Expected Outcome)

### 成功运行场景 (无变化)
- 只有 `stdout` 输出 `<memocli-result ... result="success">`。

### 参数错误场景 (优化后)
- **控制台**:
  ```text
  error: the following arguments are required: -e/--entity
  (See XML output for full help and usage)
  
  <memocli-result ... result="failed">
    ...
    <error-detail>
       错误原因: the following arguments are required: -e/--entity
       memocli append-update 的用法帮助供参考:
       usage: ...
    </error-detail>
  </memocli-result>
  ```
- 虽然 `stderr` 中依然有错误简述（这是为了保险），但耗费 Token 巨大的 `help` 信息被严格限制在了 XML 内部，且只有一份。

---

## 5. 结论

这个方案保留了 `stderr` 作为错误触发的“物理信号”，同时将大量的“认知负载”（帮助文档）收容在 XML 内部，既解决了冗余问题，又消除了您的安全顾虑。
