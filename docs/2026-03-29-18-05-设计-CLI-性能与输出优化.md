# 2026-03-29-18-05-设计-CLI 性能与输出优化方案

## 1. 问题分析 (Problem Analysis)

目前 Agent 在使用 `memories-off-skill` 时存在显著的 Token 浪费和交互效率低下的问题：

### 1.1 路径与前缀冗余 (Path Redundancy)
- **现象**: 每次调用脚本时，Agent 都会生成包含完整 Python 路径和完整知识库路径的长串命令（如用户提供的例子所示）。
- **后果**: 单次输入可能超过 1000 Tokens，且极易因为 Shell 转义问题导致路径解析失败。

### 1.2 命令链与提交冗余 (Chaining & Audit Redundancy)
- **现象**: Agent 倾向于使用 `&&` 拼接多个 `append_update.py` 操作。
- **后果**: 
    - 每一个子命令都会触发一次独立的 `commit.py` 调用。
    - 输出中会出现多次重复的 `[*] 正在提交变更...` 和 `<commit_report>`。
    - 产生了大量细碎的 Git Commits，污染了提交历史。

### 1.3 输出噪音 (Output Noise)
- **现象**: `stats.py`, `inspect_schema.py` 等工具同时输出 Plain Text（供人看）和 XML（供模型看）。
- **后果**: 内容高度重复，浪费了近一倍的 Token 空间。

## 2. 优化方案 (Proposed Solutions)

### 2.1 引入快捷启动指令 (Launcher / Alias)
- **方案**: 在 `scripts/` 目录下建立一个通用的 `run.py` 或利用 `SKILL.md` 引导 Agent 先 `export PYTHONPATH` 或 `cd`。
- **更优选**: 更新 `SKILL.md` 准则，要求 Agent 使用**相对路径**，并利用环境变量简化 `--path` 参数。

### 2.2 支持批量操作模式 (Batch Execution)
- **方案**: 升级 `append_update.py` 或引入 `batch_ops.py`。
- **细节**: 允许通过一个 JSON 字符串或临时文件一次性传入多个更新指令。
- **收益**: 仅执行一次 Git Commit，生成一份汇总的 XML Report。

### 2.3 实施“模型优先”输出协议 (Agent-First Output)
- **方案**: 更改输出方式，以「给 agent 看」优先。
- 原则：   
    - 整个输出包裹在一个 xml 块中，包含多个 child 元素。
    - 第一个元素是此次调用的参数
    - 第二个元素是整体是否成功
    - 后续元素是具体的日志或错误信息。
    - 如果有错误，则给出错误的描述，以及「应该停下来」或者「应该如何重试」的建议。

### 2.4 整合提交逻辑
- **方案**: 通过一个 batch 模式，允许一次性提交多个更新。
- batch 模式创建多个调用，然后输出。
- **细节**: 如果是批量操作，自动将理由整合（如 `Batch update: [Reason1], [Reason2]`）。

## 3. 实施计划 (Implementation Plan)

1.  **通用性改造**: 为 `schema_define.py` 增加一个基础的 `ScriptBase` 类，统一处理 XML 包裹逻辑。
2.  **脚本升级**。
3.  **规范更新**: 更新 `SKILL.md`，教授 Agent 如何进行“一次性”的高效调用。
