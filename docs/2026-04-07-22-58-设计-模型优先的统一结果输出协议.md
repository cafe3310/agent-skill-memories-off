# 设计文档: 模型优先的统一结果输出协议 (Agent-First Output Protocol)

## 1. 背景与目标 (Context & Goals)

目前 `memocli` 子命令的输出采用「人类可读 Plain Text + 模型可读 XML」的混合模式。虽然兼顾了两方，但在 Agent 协作流中存在以下问题：
1. **冗余消耗**: 相同的信息被输出了两次，浪费了 Token。
2. **结构松散**: XML 报告分布在 Plain Text 之后，Agent 需要在大量非结构化文本中寻找结构化数据。
3. **审计缺失**: 缺乏对操作「理由 (Reason)」的一等公民支持，无法在输出中直接体现审计背景。

本任务旨在定义并实现一套**统一、紧凑且模型优先**的输出协议。

---

## 2. 核心协议定义 (Protocol Definition)

所有 `memocli` 子命令的输出必须包裹在唯一的 `<memocli-result>` 根元素中，其具体结构如下：

```xml
<memocli-result subcommand="子命令名" reason="操作理由" result="success|failed">
  <source-sh>原始的完整命令行（注意，必须是自动注入或展开过的)</source-sh>
  <content>
    这里是原本人类可读的执行结果内容（例如：找到了 5 个实体）。
    如果是查询类操作，结构化的数据也可以嵌套在这里。
  </content>
  <!-- 仅当 result="failed" 时出现 -->
  <error-detail>
    具体的错误描述，以及针对 Agent 的修复建议。
  </error-detail>
</memocli-result>
```

### 2.1 关键字段说明
- **subcommand**: 当前执行的指令名（中划线风格）。
- **reason**: 从 `--reason` 或 `-r` 参数中捕获的操作理由。如果没有提供，默认为 `none`。
- **result**: 整体执行状态，成功为 `success`，失败为 `failed`。
- **source-args**: 用于回溯调用的参数，方便 Agent 确认自己发出的指令是否被正确解析。

---

## 3. 技术实现方案 (Technical Implementation)

### 3.1 增强 ScriptBase
在 `schema_define.py` 的 `ScriptBase` 中进行以下调整：

1. **统一参数注入**: 在 `_init_parser` 中为所有子命令自动增加 `-r/--reason` 参数。
2. **重构 finalize**: 
    - 废弃原有的 `<{action}_report>` 格式。
    - 实现全新的 `finalize(success, result_content, error_msg=None)` 方法。
    - 所有输出内容统一使用 `sys.stdout` 输出唯一的 XML 块。
3. **捕获结果流**: 子类不再直接 `print`，而是通过 `add_result_line` 或在 `run` 的最后返回结果字符串。

### 3.2 错误拦截
- `CustomArgumentParser.error` 也需要适配该 XML 格式，确保参数验证失败时的输出也能被 Agent 正确解析。

## 4. 迁移示例 (Migration Example)

以 `memocli stats` 为例：

**之前**:
```
[*] 正在分析知识库...
...人类报告...
<library_report total_entities="10">...</library_report>
```

**之后**:
```xml
<memocli-result subcommand="stats" reason="定期巡检" result="success">
  <source-sh>memocli stat --path=".." --init-xxx-yyy="vvv"</source-sh>
  <content>
    总计实体数: 10
    [实体类型分布]
    - 人物: 5
    - 概念: 5
  </content>
</memocli-result>
```

---

## 5. 预期收益 (Expected Benefits)

1. **极致压缩**: 消除重复输出，单次交互 Token 消耗预计降低 20%-40%。
2. **零解析成本**: Agent 始终只需要解析一个固定的根元素即可获得全量上下文。
3. **审计闭环**: 操作理由与执行结果在同一个 XML 块中强绑定，便于构建可回溯的开发日志。
