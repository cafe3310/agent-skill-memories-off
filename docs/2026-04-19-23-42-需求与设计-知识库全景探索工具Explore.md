# 需求与设计: 知识库全景探索工具 (Explore)

## 1. 背景与目标
在《其他Agent的优化建议》中提到，`inspect_schema` 作为采样工具不适合进行精确治理。但随着项目的演进，我们发现 Agent 在刚接触一个知识库时，最需要的是一个能够**一键获取项目全貌上下文**的“开眼”工具。

因此，决定将 `inspect_schema` 重构升级为 `explore` 命令。它的核心定位是：“在任务开始时，获取第一批最关键的全局和架构信息”。它不再是一个简单的 Schema 统计器，而是知识库的超级入口。

## 2. 核心功能设计 (Explore Command)

默认情况下执行 `memocli explore --path .`，输出将包含以下 6 个结构化信息块：

### 2.1 知识库元信息与路径
- 直接打印 `meta.md` 的绝对路径，方便 Agent 随时知晓当前工作锚点。

### 2.2 `meta.md` 核心指南
- 输出 `meta.md` 的内容。
- **限制**: 默认最大输出 500 行，超出部分被截断，并提示 Agent “使用 `--full` 参数查看全文”。

### 2.3 核心实体导览 (从 meta 提取)
- 扫描 `meta.md` 正文中的所有 WikiLinks (`[[实体名]]`)。
- 提取并读取这批被提及的关键实体的 Frontmatter 元数据。
- **限制**: 最多展示 20 个实体的元数据，超出部分被截断并提示。
- **价值**: 手册中提及的实体通常是该知识库的“架构基石”或“入口实体”，提供这些能帮助 Agent 迅速建立图谱意识。

### 2.4 全局实体类型分布 (Entity Types)
- **全量扫描**: 遍历所有实体文件，精确统计。
- **输出**: 列出每种 `entity type` 及其拥有的实体数量，按数量降序排列。
- **限制**: 最大展示 200 种，超出截断并提示。

### 2.5 全局关系契约分布 (Entity Relations)
- **全量扫描**: 统计所有实体中存在的出站关系 (`relation as *`)。
- **输出**: 列出每种关系谓词（Predicate）及使用该关系的实体数量，按数量降序排列。
- **限制**: 最大展示 200 种，超出截断并提示。

### 2.6 CLI 帮助清单
- 在输出末尾，通过内部调用（不使用 subprocess 而是解析或者打印机制）直接附带 `memocli --help` 的完整内容。
- **价值**: Agent 刚刚被激活时，可以一次性了解当前知识库的结构**以及**自身能够使用的全量可用工具清单，彻底解决上下文冷启动问题。

## 3. 参数设计
- `--full`: 覆盖所有的输出限制（显示完整的 `meta.md`，显示所有被提取的 WikiLink 实体的完整 Frontmatter，不再截断 Types 和 Relations 的统计）。

## 4. 实施步骤
1. 删除旧的 `inspect_schema.py`。
2. 创建新的 `commands/explore.py` 并实现上述 6 大功能块。
3. 在 `utility/schema_define.py` 或脚本内部补充对 `meta.md` WikiLinks 的提取逻辑。
4. 更新测试库并执行验证。
