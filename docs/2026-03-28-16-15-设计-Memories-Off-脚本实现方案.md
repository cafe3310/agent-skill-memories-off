# 2026-03-28-16-15-设计-Memories-Off-脚本实现方案

## 1. 背景 (Background)
在 `memories-off-skill` 转向 Git 驱动架构后，需要一套标准化的、确定性强的脚本工具集，以支持 Agent 执行知识库的初始化、观察、审计及自动修复。这套脚本将作为 Skill 的核心资源，提升 Agent 在处理复杂文件操作时的可靠性和 Token 效率。

## 2. 设计原则 (Design Principles)
- **原子化 (Atomic)**: 每个脚本专注解决一个特定维度的任务。
- **Git 语义化 (Git-Semantic)**: 深度集成 Git 流程，确保每一笔变更都具备可追溯的 Reason。
- **LLM 友好 (LLM-Friendly)**: 脚本输出统一采用结构化的 XML/JSON 报告，屏蔽冗长的系统堆栈，提供清晰的成功/失败反馈。
- **确定性 (Deterministic)**: 脚本内部处理复杂的逻辑（如正则替换、全量扫描），Agent 仅需传递核心参数。

## 3. 核心脚本清单 (Script Inventory)

### 3.1 初始化仓库 (`init.cjs`)
- **职能**: 将指定目录转变为符合规范的 `memories-off` 知识库。
- **输入**: `libraryPath`, `libraryName` (用于 meta.md)。
- **逻辑**:
    1. 执行 `git init`。
    2. 递归创建 `entities/`, `trash/` 目录。
    3. 写入初始化的 `meta.md` 手册模板。
    4. 执行 Initial Commit: `git add . && git commit -m "init: 初始化知识库"`。
- **输出**: XML 报告，确认目录结构和 Git 状态。

### 3.2 仓库概要统计 (`stats.cjs`)
- **职能**: 提供知识库的全局资产视图，辅助 Agent 建立上下文。
- **输入**: `libraryPath`。
- **逻辑**:
    1. 统计 `entities/` 下的总文件数。
    2. 解析所有实体的 Frontmatter，统计 `entity type` 的频率分布。
    3. 提取 `meta.md` 中的 Schema 规则摘要。
    4. 读取最近 N 条 Git Commit 日志作为活跃动态。
- **输出**: 包含总数、类型分布、活跃实体列表的 JSON/XML 报告。

### 3.3 变更原子提交 (`commit.cjs`)
- **职能**: 封装 Mutation 操作后的标准提交协议。
- **输入**: `action`, `target`, `reason`。
- **逻辑**:
    1. 验证目标文件是否存在。
    2. 检查是否有未暂存的变更。
    3. 执行 `git add <target>`。
    4. 执行 `git commit -m "[action] [target]: [reason]"`。
- **输出**: 提交哈希及状态确认。

### 3.4 仓库审计与自动修复 (`audit-fix.cjs`)
- **职能**: 维护库内引用一致性，纠正不规范的元数据。
- **输入**: `libraryPath`, `--fix` (可选开关)。
- **逻辑**:
    1. **孤儿关系检测**: 扫描所有 `relation as`，找出指向已不存在实体的条目。
    2. **WikiLinks 校验**: 检测正文中的 `[[...]]` 是否失效。
    3. **Schema 校验**: 检查 `entity type` 是否唯一且符合 `meta.md` 定义。
    4. **修复模式**: 自动删除失效关系，或在失效 WikiLinks 旁标注 `(broken)`。
- **输出**: 详细的问题清单，包含文件名、行号及建议修复方案。

### 3.5 实体迁移与同步 (`rename.cjs`)
- **职能**: 安全地重命名实体，并自动更新全局引用。
- **输入**: `oldName`, `newName`, `reason`。
- **逻辑**:
    1. 执行 `git mv`。
    2. 全局扫描并替换所有实体中的 WikiLinks 和 `relation as` 引用值。
    3. 调用 `commit.cjs` 记录迁移历史。
- **输出**: 受影响的文件列表及更新计数。

## 4. 技术选型 (Tech Stack)
- **Runtime**: Node.js (CommonJS)，确保与 Gemini CLI 环境的高度兼容。
- **Dependencies**: 仅使用 Node 核心库 (`fs`, `path`, `child_process`)，减少外部依赖。

## 5. 后续步骤 (Next Steps)
1.  **逐一实现**: 按照 `init -> stats -> commit -> audit-fix` 的顺序进行开发。
2.  **集成测试**: 在 `template-library` 上验证脚本的正确性。
3.  **Skill 注册**: 将这些脚本作为 `available_resources` 注册进 `memories-off-skill/SKILL.md`。
