# 开发日志：实体命名契约解耦与规范统一 (Naming-Decoupling)

## 1. 概述 (Overview)
本任务解决了“实体文件名是否包含类型前缀”的歧义问题。通过确立“文件名即实体名”的原则，彻底将物理存储路径与元数据（Type）解耦，降低了系统复杂度和认知成本。

## 2. 核心原则 (Principle)
**“文件名即实体名”**：
- 实体 `b` 的类型是 `a`，则文件名就是 `b.md`（而非 `a-b.md`）。
- 类型信息完全交由文件内部的 `entity type` 元数据维护。

## 3. 变更详情 (Actions)

### 3.1 基础设施更新
- **`init.py`**: 更新了 `meta.md` 手册模板。删除了“建议使用 [类型]-[名称]”的陈旧描述，改为“文件名即实体名”。
- **`create_entity.py`**: 更新了工具说明与示例。现在示例引导用户直接使用 `张三` 而非 `人物-张三` 作为名称。

### 3.2 规范的一致性
- 所有的读取、更新、合并工具现在都遵循这一原则：`--entity` 对应的就是物理文件名，无需进行额外的类型拼接猜测。

## 4. 改进效益 (Benefits)
1. **零冗余**: 类型变更只需修改文件内容，无需执行文件重命名（Rename）。
2. **零歧义**: 开发者和 Agent 不再需要纠结“是由我拼前缀，还是工具帮我拼”。
3. **更短的路径**: 文件名更加简洁，降低了文件系统操作的复杂度。

## 5. 结论 (Conclusion)
至此，`memories-off` 技能集的命名契约已从“隐式耦合”进化为“显性解耦”，确立了清晰的单一事实来源。
