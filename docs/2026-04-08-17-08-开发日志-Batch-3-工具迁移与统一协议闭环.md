# 开发日志：Batch 3 工具迁移与统一协议闭环 (2026-04-08)

在完成 Batch 1 (检索) 和 Batch 2 (核心修改) 的协议适配后，本项目进入了 Batch 3 的攻坚阶段。本阶段的目标是重构具有复杂副作用或系统级交互的工具，包括初始化、审计修复、跨实体合并以及图谱可视化，确保全库工具集在 `memocli` 包装下表现出高度的一致性、鲁棒性和对 Agent 的友好度。

## 3. 核心变更 (Actions)

### 3.1 系统维护工具重构
- **`init.py` (InitScript)**:
    - 将过程指令化。在创建 `meta.md` 和初始化 Git 仓库的同时，将每一步的成功/失败信号捕获至 `<content>` 节点。
    - 允许路径不存在时自动递归创建，增强了“冷启动”体验。
- **`commit.py` (CommitScript)**:
    - 作为审计核心，强制要求 `--reason/-r` 参数。
    - 自动将 Git 的 stdout/stderr 封装进 XML 报告，使 Agent 能够直接感知版本控制的状态，而无需解析凌乱的 shell 输出。

### 3.2 复杂副作用逻辑重构
- **`merge_entities.py` (MergeEntitiesScript)**:
    - 实现了多源合并逻辑。自动提取 Body（剔除元数据）、追加到目标实体，并触发全库范围内的 WikiLinks 与 Relations 重定向。
    - 整合了自动删除源文件与 Git 提交，实现了“原子化”的合并操作。
- **`merge_types.py` (MergeTypesScript)**:
    - 优化了全库元数据批量更新逻辑。
    - 同步维护 `meta.md` 中的类型定义列表，确保 Schema 的单一事实来源。

### 3.3 审计与可视化增强
- **`audit_fix.py` (AuditFixScript)**:
    - 确立了“先审计，后修复”的模式。在 `--fix` 模式下强制要求理由。
    - 改进了错误定位，输出格式适配 `<error-detail>`，方便 Agent 根据建议进行手术式修复。
- **`graph.py` (GraphScript)**:
    - 虽然是交互式 HTML 工具，但通过继承 `ScriptBase` 实现了协议闭环。
    - 临时文件路径和浏览器启动状态现在通过标准的 XML 结果上报，Agent 可以据此确认可视化是否已成功送达用户。

## 4. 交付结果 (Outcome)

- **全量适配**: `memories-off` 核心脚本库已 100% 完成 `ScriptBase` 迁移。
- **协议闭环**: 
    - `stdout`: 始终为唯一的 `<memocli-result>` XML。
    - `stderr`: 仅保留极简单行信号。
    - `exit code`: 精确反映操作成败。
- **Agent 友好**: 所有变动理由均已通过 `--reason` 强制上报，建立了完整的知识变更审计链。
