# 开发日志: `create-entity` 增强与更新块逻辑收拢

## 1. 背景与问题
在 `2026-04-12-00-52-设计-其他Agent的优化建议.md` 中，指出当前 `create-entity` 存在一个主要缺陷：**不能直接带正文内容 (`--content`) 和别名 (`--aliases`)**，导致 Agent 新建实体往往需要 `create` + `append-update` 两步走，操作琐碎且容易因为中断导致状态残缺。

同时，为了在实体级别实现更好的“时间线溯源”和“过程追踪”，我们决定：**当使用 `create-entity` 携带 `--content` 创建实体时，其正文内容也应当被包裹进一个标准的 `UPDATE_BLOCK` 中**，而不是直接散落在正文里。

## 2. 架构重构目标
目前有关 `UPDATE_BLOCK_START ... UPDATE_BLOCK_END` 的正则匹配、构造生成等代码，分散在 `append_update.py` 和 `consolidate_updates.py` 中。
为了实现上述需求，我们需要先将更新块 (Update Block) 的逻辑抽象出来，并在 `create-entity` 中复用它。

## 3. 具体修改计划

### 3.1 步骤一：在 `schema_define.py` 中引入 `UpdateBlockManager`
新建一个专用的类或在 `MetadataParser` 中添加相关静态方法，用来集中处理缓冲更新块的生命周期：
*   **`create_block(content: str, reason: str) -> str`**:
    *   生成符合格式的带有时间戳和 `reason` 的 HTML 注释包裹块。
    *   在内部自动调用 `normalize_wikilinks`，确保内容写入前已被合规化。
*   **`extract_blocks(content: str) -> list[tuple]`**:
    *   将 `consolidate_updates.py` 中的 `NEW_BLOCK_RE` 正则提取到这里。

### 3.2 步骤二：重构 `append_update.py` 和 `consolidate_updates.py`
将这两个脚本中硬编码的正则字符串和字符串拼接逻辑，全部替换为调用 `schema_define.py` 中新封装的方法，确保一致性。

### 3.3 步骤三：增强 `create_entity.py`
*   **新增参数**:
    *   `-c, --content`: 可选，创建实体时的正文内容。
    *   `--aliases`: 可选，实体的别名（支持多个，逗号分隔）。
*   **逻辑实现**:
    *   **处理别名**: 在初始生成 metadata 时，解析 `--aliases` 并使用已有的 `add_alias` 方法注入到 metadata 中。
    *   **处理正文**:
        *   如果不传 `--content`，维持原样，留一段占位符。
        *   如果传了 `--content`，则调用底层的 `create_block(content, reason)` 并在实体头部标题下方写入该块。

## 4. 预期收益
1.  新建实体一气呵成，省去后续追写正文和补充别名的独立 tool call，极大减少 token 消耗。
2.  任何新增业务内容全部结构化地进入 `UPDATE_BLOCK` 格式，与后续的 `consolidate-updates` (梦境整理) 形成天然闭环。
3.  收拢冗余的正则匹配，使代码库更加整洁。
